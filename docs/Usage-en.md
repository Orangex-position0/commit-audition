## Interactive Generation

### Basic Usage

```bash
commit-audition
```

Launches an interactive Q&A to generate a commit message step by step.

### Step-by-step Walkthrough

#### Step 1: Select Commit Type

```
? 选择 commit 类型
> feat - 新功能 (feature)
  fix - 修补 bug
  docs - 文档改变
  style - 格式（不影响代码运行的变动）
  refactor - 重构
  tests - 增加测试
  chore - 构建过程或辅助工具的变动
```

Use arrow keys to select, Enter to confirm.

#### Step 2: Enter Title

```
? 输入 commit 标题 (命令式预期，首字母大写，<= 50 字符): Add user login feature
```

**Validation rules**:
- Must not be empty
- Display width must not exceed 50 characters (CJK chars count as 2 width each)
- Must not end with a period

Validation errors appear below the input prompt and require re-entry.

#### Step 3: Add Body (Optional)

```
? 是否添加正文? (y/N)
```

Selecting `Yes` opens the editing mode selector:

```
? 选择编辑方式
> Terminal Inline - 在终端中逐行输入
  Default Editor - 使用系统默认编辑器
  Custom Editor - 使用配置文件中指定的编辑器
```

See [Body Editing Modes](#body-editing-modes) for details.

#### Step 4: Link Issue (Optional)

```
? 是否添加关联 issue? (y/N)
? 输入 issue 编号: 42
```

#### Step 5: Preview and Confirm

```
── 预览 ──
feat: Add user login feature

Implemented OAuth2 flow.

#42

? 确认使用这条 commit message? (Y/n)
```

- **Confirm**: Outputs plain text commit message
- **Reject**: Asks whether to re-edit. Re-editing returns to Step 1, otherwise cancels.

### Output Example

```text
feat: Add user login feature

Implemented OAuth2 flow.

#42
```

You can copy it directly or pipe it to `git commit`:

```bash
commit-audition | xargs -0 git commit -m
```

## Validation Mode

### Validate a File

```bash
commit-audition validate .git/COMMIT_EDITMSG
```

Commonly used in CI/CD pipelines or manual checks.

### Read from Stdin

```bash
echo "feat: Add feature" | commit-audition validate -
```

Use `-` as the file path argument to read from standard input.

### Exit Codes

| Exit Code | Meaning |
|---|---|
| `0` | Validation passed |
| `1` | Validation failed (error message printed to stderr) |

### Validation Failure Examples

```bash
$ commit-audition validate .git/COMMIT_EDITMSG
标题: 长度超出限制，当前长度为 65，最大长度为 50
```

```bash
$ echo "unknown: some message" | commit-audition validate -
不合法的类型 "unknown"
  合法类型: feat, fix, docs, style, refactor, tests, chore
```

### Validation Rules

| Rule | Limit | Error Type |
|---|---|---|
| Message must not be empty | — | `Empty` |
| Must have `<type>: ` prefix | — | `MissingType` |
| Type must be valid | 7 types | `InvalidType` |
| Title must not be empty | — | `TitleError::Empty` |
| Title display width | ≤50 | `TitleError::TooLong` |
| Title must not end with period | — | `TitleError::EndsWithPeriod` |
| Body line display width | ≤72 | `BodyError::LineTooLong` |

## Git Hook Management

### Install Hook

```bash
commit-audition hook install
```

Creates a `commit-msg` hook in the current git repo's `.git/hooks/` directory.

**Effect**: Every `git commit` auto-validates the commit message. Non-compliant messages are rejected.

**Safety checks**:
- If an existing hook was generated by this tool → prompts "already installed", skips
- If an existing hook was NOT generated by this tool → errors and asks for manual inspection, never overwrites

### Uninstall Hook

```bash
commit-audition hook uninstall
```

Removes the `commit-msg` hook from the current repo.

**Safety check**: Only deletes hooks containing the `commit-audition` identifier. Never deletes user-defined hooks.

### Typical Workflow

```bash
# First time: install hook
cd your-project
commit-audition hook install

# Normal commit (hook auto-validates)
git add .
git commit -m "feat: Add new feature"
# → Validation passes, commit succeeds

git commit -m "bad message"
# → Validation fails, commit is rejected
```

## Body Editing Modes

### Terminal Inline

Enter body text line by line in the terminal. Submit an empty line to finish.

```
逐行输入正文，输入空行结束:
? 第 1 行 (留空结束): The timeout was set to 5 seconds.
? 第 2 行 (留空结束): Increased to 30 seconds for slow networks.
? 第 3 行 (留空结束):
```

**Characteristics**:
- Best for short bodies (1-5 lines)
- No need to leave the terminal
- Auto-validates line width (≤72 chars) per line

### Default Editor

Uses the editor specified by the `VISUAL` or `EDITOR` environment variable.

```bash
# Set default editor
export EDITOR=vim
# or
export VISUAL=code
```

**Characteristics**:
- Best for longer bodies
- Full editor features (syntax highlighting, search/replace, etc.)
- Comment lines (starting with `#`) are auto-filtered

### Custom Editor

Uses the editor command specified in the config file.

Requires configuration in `~/.commit-audition/config.toml`:

```toml
[editor]
command = "code --wait"
extension = "md"
```

**Characteristics**:
- Specify any editor with arguments
- Configurable temp file extension for correct syntax highlighting
- Shows config example when not configured

### Mode Comparison

| Mode | Best For | Requires Config | Leaves Terminal |
|---|---|---|---|
| Terminal Inline | Short bodies | No | No |
| Default Editor | Longer bodies | Env var needed | Yes |
| Custom Editor | Specific editor preference | Config file needed | Yes |

## Configuration

### Config File Path

```text
~/.commit-audition/config.toml
```

### Full Config Example

```toml
[editor]
command = "code --wait"    # Custom editor command (optional)
extension = "md"            # Temp file extension (default "md")
```

### Common Editor Configs

| Editor | command Value |
|---|---|
| VS Code | `"code --wait"` |
| Vim | `"vim"` |
| Neovim | `"nvim"` |
| Nano | `"nano"` |
| Sublime Text | `"subl --wait"` |
| Emacs | `"emacs"` |

### Loading Rules

- File does not exist → use defaults
- File content is invalid → use defaults
- Some fields missing → those fields use defaults

## Commit Message Convention

### Format

```text
<type>: <title>           ← Title line (required, ≤50 chars display width)
                          ← Blank line (only when body or issue present)
<body>                    ← Body (optional, each line ≤72 chars display width)
                          ← Blank line (only when both body and issue present)
#<issue>                  ← Issue reference (optional)
```

### Examples

#### Minimal

```text
feat: Add user login feature
```

#### Title + Body

```text
fix: Fix login timeout

The timeout was set to 5 seconds.
Increased to 30 seconds for slow networks.
```

#### Full

```text
feat: Add user login

Implemented OAuth2 flow.

#42
```

### Title Rules

- Use imperative mood ("Add" not "Added" or "Adds")
- Capitalize the first letter
- Must not exceed 50 chars display width (CJK chars count as 2 each)
- Must not end with a period

### Body Rules

- Each line must not exceed 72 chars display width
- Explain "why" not "what" ("what" should be in the title and code diff)
